一、本章考什么、怎么学

核心结论：这一章延续递归，但重点从“会写递归”转到“怎样让递归更聪明”。主线是：分治拆问题；贪心试图每步选最好；朴素递归会爆炸；记忆化和动态规划通过记录中间结果避免重复计算。

先分清三种策略

贪心：只看当前一步；递归：把剩余问题交给自己；动态规划：先把小问题答案填好，再拼出大问题。

动态规划不要一上来背代码

先问三个问题：状态表示什么？从哪些更小状态转移？表格按什么顺序填？

代码题看“表”

动态规划代码看起来长，但核心往往就是一个列表或二维表：记录中间结果，后面直接查。

二、分治策略与递归

1. 什么是分治策略

分治策略：把一个问题分成若干个规模更小的部分；解决每个小部分；再把小部分的结果汇总成原问题的答案。
分治不是某一个具体算法，而是一类解决问题的思路。

2. 递归为什么常常体现分治

递归三定律本身就很像分治：有基本结束条件；每次把问题规模变小；调用自身解决更小的相同问题。许多排序、查找、遍历、求值算法都会用到这一思想。

例题1：分治复杂度判断

题目思想：若使用分治算法，每次把规模为 n 的问题分解为 2 个规模为 n/2 的子问题，合并开销为 O(n)，总时间复杂度是多少？

参考答案：O(n log n)。因为每一层合并总开销约为 O(n)，问题规模每次减半，所以递归树有 log n 层，总计 O(n log n)。

三、优化问题与贪心策略

1. 什么是优化问题

优化问题：目标是找到某种“最优解”，例如最短路径、最少硬币、最大价值、最小集合等。

2. 贪心策略怎么想

贪心策略的直觉是：每一步都尽量解决“当前最大的一部分”。找零时就是先用最大面值硬币，能用多少用多少，再考虑下一种面值。

3. 找零兑换中的贪心失效

关键反例：如果硬币有 1、5、10、21、25 分，要找 63 分。贪心先用 25 分，得到 25+25+10+1+1+1，共 6 枚；但真正最优是 21+21+21，共 3 枚。

四、找零兑换：递归解法

1. 思路：试所有第一枚硬币

递归解法：为了保证找到最优解，不只盯着最大硬币，而是对每一种可用硬币都试一次：先用这一枚硬币，再递归求“剩余金额”的最少硬币数，最后取最小值。

用公式表达就是：

numCoins(change) = min(1 + numCoins(change-c))，其中 c 遍历所有不超过 change 的硬币面值。
前提说明：课程例子通常默认硬币体系里有 1 分硬币，所以任何整数金额都能找开。如果没有 1 分硬币，原来把 minCoins 初始化成 change 的写法会有风险：无解时也可能返回 change。下面统一改成更稳的写法：无解返回 None，动态规划中用 float('inf') 表示暂时不可达。

2. 代码解释

代码解释：朴素递归找零
def recMC(coinValueList, change):
    # 找零 0 分需要 0 枚硬币
    if change == 0:
        return 0

    # 基本结束条件：change 本身就是某种硬币
    if change in coinValueList:
        return 1

    minCoins = None   # None 表示目前还没有找到可行方案

    # 尝试每一种不超过 change 的硬币
    for coin in [c for c in coinValueList if c <= change]:
        subResult = recMC(coinValueList, change - coin)

        # 如果剩余金额无解，就跳过这种硬币
        if subResult is not None:
            numCoins = 1 + subResult
            if minCoins is None or numCoins < minCoins:
                minCoins = numCoins

    return minCoins

print(recMC([1, 5, 10, 21, 25], 63))  # 3
print(recMC([5, 10], 3))              # None，表示无法找零
小白理解：这段代码不是“猜最大硬币”，而是把所有可能第一步都试一遍。只要某条路最后能凑出目标金额，就参与比较；如果某条路剩余金额无解，就返回 None 并跳过。这样既能找最优解，也不会把“无解”误判成需要 change 枚硬币。

3. 为什么它极其低效

同一个金额可能从不同分支反复被计算，这就是低效的根源。

五、记忆化：减少重复计算

1. 核心思想

记忆化 memoization：用一个表保存已经算过的中间结果。递归前先查表；如果表里有，直接返回；如果没有，才递归计算并把结果存入表。

2. 改进版代码

代码解释：带记忆化的 recDC
UNKNOWN = -1


def recDC(coinValueList, change, knownResults):
    # 查表成功：可能是整数，也可能是 None（表示无解）
    if knownResults[change] != UNKNOWN:
        return knownResults[change]

    if change == 0:
        knownResults[change] = 0
        return 0

    if change in coinValueList:
        knownResults[change] = 1
        return 1

    minCoins = None

    for coin in [c for c in coinValueList if c <= change]:
        subResult = recDC(coinValueList, change - coin, knownResults)
        if subResult is not None:
            numCoins = 1 + subResult
            if minCoins is None or numCoins < minCoins:
                minCoins = numCoins

    # 所有硬币都尝试完后，统一把最终结果写入记忆表
    knownResults[change] = minCoins
    return minCoins

memo = [UNKNOWN] * 64
print(recDC([1, 5, 10, 21, 25], 63, memo))  # 3
print(recDC([5, 10], 3, [UNKNOWN] * 4))     # None
为什么这样写更标准：循环里是在“尝试某一种硬币”。只有把所有可用硬币都试完以后，minCoins 才是当前 change 的最终结果，所以更适合在 for 循环结束后统一写入 knownResults[change]。这里用 UNKNOWN=-1 表示“还没算过”，用 None 表示“已经确认无解”，这样查表时不会混淆。

例题2：记忆化为什么能提速？

题目思想：朴素递归找零会重复计算许多金额，比如 15 分。记忆化为什么能减少递归调用？

参考答案：因为第一次算出 15 分的最优解后，会把结果存入 knownResults[15]。下次再遇到 15 分，直接查表返回，不再展开一整棵递归子树。

3. 它和动态规划还不是一回事

区别：记忆化通常仍然是递归，只是在递归中加缓存；动态规划通常从最小规模开始按顺序填表，是更有条理的自底向上做法。

六、动态规划：找零兑换

1. 动态规划从哪里开始

动态规划思路：从最简单的 1 分钱找零开始，逐步计算 2 分、3 分……一直到目标金额。每一步都依赖更小金额的最优解。

例如要算 11 分，硬币为 1、5、10。可以考虑最后一枚硬币分别是 1、5、10：

最后一枚硬币剩余金额查表结果总硬币数
1 分10 分10 分最优为 1 枚1 + 1 = 2
5 分6 分6 分最优为 2 枚1 + 2 = 3
10 分1 分1 分最优为 1 枚1 + 1 = 2

取最小值，所以 11 分至少需要 2 枚硬币。

2. 代码解释

代码解释：dpMakeChange
def dpMakeChange(coinValueList, change, minCoins):
    INF = float('inf')
    minCoins[0] = 0

    for cents in range(1, change + 1):
        coinCount = INF       # 先假设当前金额不可达

        for coin in [c for c in coinValueList if c <= cents]:
            # 假设最后一枚硬币是 coin
            if minCoins[cents - coin] + 1 < coinCount:
                coinCount = minCoins[cents - coin] + 1

        minCoins[cents] = coinCount

    if minCoins[change] == INF:
        return None           # 没有任何组合能凑出 change
    return minCoins[change]

amount = 63
minCoins = [float('inf')] * (amount + 1)
print(dpMakeChange([1, 5, 10, 21, 25], amount, minCoins))  # 3
print(dpMakeChange([5, 10], 3, [float('inf')] * 4))        # None
小白理解：minCoins[x] 表示“凑出 x 分最少要几枚硬币”。算 x 时，只需要查更小的 x-coin，这些答案已经在前面算好了。这里用 float('inf') 表示“暂时凑不出来”，最后如果仍然是无穷大，就返回 None。

例题3：分治和动态规划是否都分解子问题？

题目思想：“分治法和动态规划法都运用了将问题分解为规模较小的子问题的思想。”这句话对吗？

参考答案：对。二者都把原问题拆成规模更小的子问题；区别在于，动态规划特别强调记录并复用重叠子问题的答案，避免同一个子问题反复计算。

例题4：动态规划为什么要求最优子结构？

题目思想：为什么找零兑换可以用动态规划？

参考答案：因为一个金额的最优解可以由更小金额的最优解推出。例如最后拿一枚 5 分硬币时，剩余金额 change-5 必须也用最少硬币，否则原问题就不可能最优。

七、记录硬币组合与回溯

1. 为什么还要记录选择的硬币

前面的动态规划只能告诉我们“最少几枚”，但没有告诉我们“是哪几枚”。要输出硬币组合，就需要在填表时同步记录：当前金额最优时选择了哪枚硬币。

核心做法：用 coinsUsed[cents] 记录凑出 cents 时最后选择的那枚硬币。最后从目标金额开始，不断减去这枚硬币，就能反推出整套组合。
代码解释：同时记录 coinsUsed
def dpMakeChange(coinValueList, change, minCoins, coinsUsed):
    INF = float('inf')
    minCoins[0] = 0
    coinsUsed[0] = 0

    for cents in range(1, change + 1):
        coinCount = INF
        newCoin = None

        for coin in [c for c in coinValueList if c <= cents]:
            if minCoins[cents - coin] + 1 < coinCount:
                coinCount = minCoins[cents - coin] + 1
                newCoin = coin

        minCoins[cents] = coinCount
        coinsUsed[cents] = newCoin

    if minCoins[change] == INF:
        return None
    return minCoins[change]


def printCoins(coinsUsed, change):
    if coinsUsed[change] is None:
        print('无法找零')
        return

    cents = change       # cents 表示“当前还剩多少金额没凑齐”
    while cents > 0:
        thisCoin = coinsUsed[cents]   # thisCoin 才是本轮选择的硬币面值
        if thisCoin is None:
            print('无法找零')
            return
        print(thisCoin)
        cents = cents - thisCoin      # 剩余金额减少
为什么这里不用 coin 当循环变量：cents 表示“当前还剩多少金额没凑齐”，thisCoin 才表示“这一步选择的硬币面值”。这样命名更清楚，不会把“剩余金额”和“硬币面值”混在一起。
为什么 coinsUsed 也要能记录无解：如果某个金额根本凑不出来，coinsUsed[cents] 就保持为 None。回溯输出硬币组合时，一看到 None，就知道不能继续减，否则会进入错误结果或死循环。

八、博物馆大盗问题

1. 问题是什么

大盗有一个容量有限的背包，面前有若干宝物，每件宝物有重量和价值。目标是在不超过背包容量的前提下，让总价值最大。

宝物重量价值

348

2. 状态 m(i, W) 怎么理解

状态定义：m(i, W) 表示：只考虑前 i 件宝物，并且背包容量不超过 W 时，能得到的最大价值。

第 i 件宝物只有两种选择：不拿，或者拿。

不拿第 i 件

价值就是 m(i-1, W)。

拿第 i 件

价值就是 m(i-1, W-Wi) + vi，前提是容量够。

状态转移：m(i,W)=max(m(i-1,W), m(i-1,W-Wi)+vi)。

3. 代码思路

代码解释：0/1 背包 DP 思路
def museum_thief(items, capacity):
    # items 形如 [(weight, value), ...]
    n = len(items)
    dp = [[0] * (capacity + 1) for _ in range(n + 1)]

    for i in range(1, n + 1):
        weight, value = items[i - 1]
        for w in range(1, capacity + 1):
            # 方案1：不拿第 i 件
            dp[i][w] = dp[i - 1][w]

            # 方案2：拿第 i 件，前提是装得下
            if weight <= w:
                dp[i][w] = max(dp[i][w], dp[i - 1][w - weight] + value)

    return dp[n][capacity]

items = [(2,3), (3,4), (4,8), (5,8), (9,10)]
print(museum_thief(items, 20))
