一、本章考什么、怎么学

核心结论：这一章是后面所有代码题、复杂度题的地基。你要学会把“程序跑了多久”转换成“随着问题规模 n 变大，操作次数怎么增长”。

1. 本章知识地图

同一问题→多种算法→比较资源消耗→抽象成 T(n)→写成大O
模块你要懂什么常见考法
算法分析比较的是算法的资源消耗，不只是程序写得好不好看。概念判断、为什么要分析算法。
运行时间检测可以用 time / timeit 实测，但实测受机器、语言、环境影响。代码理解、实验结果解释。
大O表示法忽略常数和低阶项，只看增长最快的主导项。给 T(n) 求 O；给代码求 O。
变位词四种解法同一个问题可有 O(n²)、O(n log n)、O(n!)、O(n) 等不同算法。算法比较、空间换时间。
list/dict 性能list 按下标访问快；dict 按 key 查找快；list 中部删除/查找值可能慢。选择合适数据类型、分析 Python 操作。

2. 小白学习法

先看循环层数

一个循环通常先想到 O(n)；两层嵌套循环通常先想到 O(n²)。但要注意循环是否真的都跑满。

再看隐藏操作

sort()、in、pop(0)、列表拼接 + 都不是“免费”的。代码短不代表复杂度低。

最后看主导项

T(n)=5n²+27n+1005，当 n 很大时，n² 决定增长速度，所以是 O(n²)。

二、算法分析：比程序，还是比算法？

1. 程序和算法不是一回事

图示理解：同一个算法可以被不同人写成不同程序；同一个问题也可以有不同算法。
核心结论：算法是解决问题的步骤；程序是用某种编程语言实现这个算法。算法分析主要比较算法本身的资源消耗。

2. 两段代码看起来不同，算法可能一样

课件用“从 1 累加到 n”的例子说明：一段代码变量名清楚，另一段代码变量名很怪、还有无用代码，但它们本质上都在做同一个循环累加算法。

写得正常的版本
def sumOfN(n):
    theSum = 0
    for i in range(1, n + 1):
        theSum = theSum + i
    return theSum
变量名很差，但算法本质一样
def foo(tom):
    fred = 0
    for bill in range(1, tom + 1):
        barney = bill
        fred = fred + barney
    return fred
怎么判断？不要被变量名骗了。两段代码都做了 n 次加法，都是循环累加，算法资源消耗属于同一类。

3. 什么是算法分析

算法分析就是从计算资源消耗角度评判算法。资源主要有两类：时间和空间。在这章中，重点先看时间。

易错点：代码风格、可读性很重要，但它不是算法分析最核心的问题。算法分析主要问：问题规模变大时，执行步骤增长得有多快？

三、运行时间检测与它的局限

1. 用 time 模块测运行时间

最直接的办法：算法开始前记录时间，结束后再记录时间，相减得到运行时间。

代码解释：用 time 计时
import time

def sumOfN2(n):
    start = time.time()       # 记录开始时间
    theSum = 0
    for i in range(1, n + 1):
        theSum = theSum + i
    end = time.time()         # 记录结束时间
    return theSum, end - start

print(sumOfN2(10000))
观察结论：循环累加中，n 变成 10 倍，循环次数也大约变成 10 倍，运行时间也会大致变长。

2. 公式法为什么更快

代码解释：无迭代公式法
def sumOfN3(n):
    return n * (n + 1) / 2

这个算法不需要循环，不管 n 是一万、一百万还是一亿，代码都只做固定数量的计算。因此它的运行时间几乎不随 n 变化。

3. 实测运行时间有什么问题

  • 同一算法用不同语言写，运行时间不同。
  • 同一程序在不同机器上跑，运行时间不同。
  • 同一机器不同时刻负载不同，结果也会波动。
所以：我们需要一个和机器、语言、运行时段尽量无关的指标，这就引出“大O表示法”。

四、大O表示法

1. 问题规模 n 是什么

问题规模就是影响执行时间的主要输入大小。比如“求 1 到 n 的和”，规模就是 n；“判断两个长度相等的字符串是否为变位词”，规模就是字符串长度 n。

2. T(n) 与大O

T(n)：基本操作随着问题规模 n 变化的执行次数函数。
大O：只保留 T(n) 中增长最快的主导部分，忽略常数和低阶项。
图示不是精确函数图，而是帮助理解：常数、对数、线性、平方的增长速度差异。

3. 怎么从 T(n) 得到大O

T(n)怎么化简大O
1+n常数 1 越来越不重要，保留 nO(n)
5n²+27n+1005保留最高阶 n²，去掉系数 5O(n²)
2n+26保留 n，去掉常数和系数O(n)

4. 常见复杂度从小到大

O(1)常数
O(log n)对数
O(n)线性
O(n log n)对数线性
O(n²)平方
O(n³)立方
O(2ⁿ)指数
易错点：大O不是精确运行时间，而是增长趋势。例如 O(n) 不代表一定比 O(1) 慢一秒，只代表 n 变大时它会随 n 线性增长。

五、从代码判断复杂度

1. 基本方法

做题步骤：先确定 n 是什么 → 找最频繁执行的语句 → 数它执行多少次 → 保留最高阶。

2. 典型代码：两层嵌套循环

代码解释：从赋值次数得到 T(n)
a = 5
b = 6
c = 10

for i in range(n):
    for j in range(n):
        x = i * i
        y = j * j
        z = i * j

for k in range(n):
    w = a * k + 45
    v = b * b

d = 33
怎么数：前面 3 个赋值是常数；双重循环里面 3 条赋值，每条执行 n² 次，所以是 3n²；后面单循环 2 条赋值，每条执行 n 次，所以是 2n；最后还有 1 个赋值。
因此 T(n)=3n²+2n+4，大O是 O(n²)。

3. 最好、最坏、平均情况

有些算法的运行时间不仅取决于 n，还取决于数据本身。比如查找一个元素，可能第一下就找到，也可能到最后才找到。

情况意思做题时怎么用
最好情况数据刚好很有利，执行次数最少通常不是主要评价依据。
最坏情况数据最不利，执行次数最多考试常问，表示保守保证。
平均情况考虑一般数据分布下的平均表现更接近日常主流性能。

4. 例题：从代码判断复杂度

例题：嵌套循环的时间复杂度

题目思想：已知一个列表 a 的长度为 n，判断下面代码的时间复杂度。

length = 1
for i in range(n - 1):
    for j in range(i + 1, n):
        if a[j - 1] > a[j]:
            break
    if length < j - i + 1:
        length = j - i + 1

前面已经学过：问题规模 n、嵌套循环、最坏情况分析、1+2+...+n 的数量级。

参考答案：最坏情况下内层循环不会很快 break，外层第 1 次内层最多跑 n-1 次，第 2 次最多跑 n-2 次……总比较次数约为 (n-1)+(n-2)+...+1，所以时间复杂度为 O(n²)。

六、变位词判断四种解法

1. 什么是变位词

变位词：两个词的字母可以通过重新排列互相得到。例如 heart 和 earth，python 和 typhon。

为了简化，课程假设两个词只由小写字母构成，而且长度相等。目标是写一个布尔函数：是变位词返回 True，否则返回 False。

2. 解法1：逐字检查 O(n²)

把 s1 的每个字符拿去 s2 中找，找到就标记为 None，避免重复使用同一个字符。

代码解释：逐字检查
def anagramSolution1(s1, s2):
    alist = list(s2)          # 字符串不可变，先转成列表
    pos1 = 0
    stillOK = True

    while pos1 < len(s1) and stillOK:
        pos2 = 0
        found = False
        while pos2 < len(alist) and not found:
            if s1[pos1] == alist[pos2]:
                found = True
            else:
                pos2 += 1
        if found:
            alist[pos2] = None   # 打勾：这个字符已经被配对
        else:
            stillOK = False
        pos1 += 1
    return stillOK
为什么是 O(n²)：外层要看 s1 的 n 个字符；每个字符最坏都要在 s2 里找一遍。总比较次数接近 1+2+...+n，数量级是 O(n²)。

3. 解法2：排序比较 O(n log n)

把两个字符串都排序，再逐位比较。排序通常是主导步骤。

代码解释：排序比较
def anagramSolution2(s1, s2):
    alist1 = list(s1)
    alist2 = list(s2)
    alist1.sort()
    alist2.sort()

    pos = 0
    matches = True
    while pos < len(s1) and matches:
        if alist1[pos] == alist2[pos]:
            pos += 1
        else:
            matches = False
    return matches
注意：看上去只有一个 while 循环，但前面的 sort() 不是免费的。不能只数自己看得见的循环。

4. 解法3：暴力全排列 O(n!)

生成 s1 的所有排列，再看 s2 是否在里面。这个思路很直观，但极其慢。长度为 20 的字符串有 20! 种排列，课件指出即使用很快的速度处理，也可能需要近 8 万年。

考试判断：暴力法通常可以作为“正确但不可行”的反例，说明算法不能只看思路是否完整，还要看复杂度。

5. 解法4：计数比较 O(n)

给 26 个小写字母各建一个计数器，分别统计两个词中每个字母出现次数，最后比较两个计数器是否完全一样。

代码解释：计数比较
def anagramSolution4(s1, s2):
    c1 = [0] * 26
    c2 = [0] * 26

    for ch in s1:
        pos = ord(ch) - ord('a')
        c1[pos] += 1

    for ch in s2:
        pos = ord(ch) - ord('a')
        c2[pos] += 1

    return c1 == c2
为什么是 O(n)：前两个循环各扫一遍字符串，共 2n；最后比较 26 个计数，26 是常数。所以 T(n)=2n+26，大O是 O(n)。
空间换时间：计数法要额外用两个计数器列表，所以比前几种方法占用更多空间，但速度更快。这就是常见的时间与空间权衡。

七、Python list 的性能

1. list 的常见操作

list 是 Python 最常用的内置数据类型之一。它适合按下标访问元素，比如 a[i]。

操作复杂度直觉原因
a[i]、a[i] = vO(1)按下标随机访问，位置可以直接算出来。
append(v)通常 O(1)在末尾追加，最常用，Python 做了优化。
lst = lst + [v]O(n+k)要生成新列表并复制原列表和新列表内容。
pop()O(1)从末尾删，不需要移动前面的元素。
pop(0) 或中间删除O(n)删掉后，后面的元素要整体往前挪。
x in listO(n)通常要从头到尾一个个找。

2. 为什么 append 比 + 快很多

慢：循环中反复拼接
lst = []
for i in range(n):
    lst = lst + [i]  # 每次都复制旧列表
快：append 追加
lst = []
for i in range(n):
    lst.append(i)   # 直接在末尾追加
理解：+ 不是“在原地加一下”，而是创建新列表；循环里不断 + 会反复复制，代价很大。append 是专门为尾部追加优化的。

3. pop() 和 pop(0) 为什么差别巨大

这张图是本章 list 性能的关键：删末尾不用移动，删开头/中间要搬家。

4. 例题：list 的中部删除为什么慢

例题：为什么 pop(0) 比 pop() 慢？

题目思想：同样是删除一个元素，为什么从末尾删除通常是 O(1)，从开头删除是 O(n)？

前面已经学过：list 按下标存放元素、pop() 删除末尾、pop(0) 删除开头，以及“开头/中间删除会导致后面元素整体前移”。

参考答案：pop() 删除列表末尾元素，不需要移动其他元素；pop(0) 删除开头元素后，后面所有元素都要整体向前移动一位，因此元素越多，移动代价越大，时间复杂度是 O(n)。

八、Python dict 的性能

1. dict 和 list 的核心差别

list：根据位置 index 找元素。
dict：根据关键码 key 找元素。

2. dict 的常用操作

操作平均复杂度解释
d[key]O(1)根据 key 直接定位。
d[key] = valueO(1)插入或更新键值对。
key in dO(1)判断 key 是否存在。
x in listO(n)和 dict 对比：list 查值通常要线性扫描。
代码解释：list in 与 dict in
nums = [10, 20, 30, 40]
print(30 in nums)       # list：可能从头扫到尾，O(n)

score = {'Tom': 90, 'Ann': 95, 'Bob': 88}
print('Ann' in score)   # dict：查 key，平均 O(1)
print(score['Ann'])     # 根据 key 取值，平均 O(1)
小白理解：list 像排队名单，要找某个名字可能从头看到尾；dict 像带索引的通讯录，知道 key 就能很快定位。

3. 为什么 x in list 是 O(n)，而 key in dict 平均是 O(1)

list 查“某个值”为什么是 O(n)

list 主要擅长的是按下标访问，比如 a[3]。但如果你问 30 in nums，Python 通常不知道 30 在第几个位置，只能从 nums[0] 开始一个个比较：第 1 个不是，看第 2 个；第 2 个不是，看第 3 个……

如果目标值刚好在开头，会很快；如果在最后，或者根本不存在，就要看完整个列表。所以最坏情况是比较 n 次，平均也会随着 n 增大而增大，因此记作 O(n)。

dict 查“key”为什么平均是 O(1)

dict 背后使用的是类似“散列表”的思想。查 'Ann' in score 时，Python 不是从第一个键开始挨个找，而是先对 key 做一次哈希计算：hash('Ann')，把 key 转成一个整数，再用这个整数快速定位到内部表中的候选位置。

定位后再检查那里的 key 是否真的是 'Ann'。如果哈希分布比较均匀，冲突被控制得较好，查找需要检查的位置数量通常近似固定，不会随着字典规模线性增长，所以平均复杂度是 O(1)。

注意说的是平均 O(1)：如果发生哈希冲突，Python 还需要额外检查；但正常情况下冲突被控制得较好，所以平均查找次数近似常数。
考试写法：x in list 查的是“值”，list 没有值到位置的直接映射，所以通常顺序扫描，O(n)；key in dict 查的是“键”，dict 可通过哈希函数快速定位，平均 O(1)。

4. 例题：频繁查找时选 list 还是 dict

例题：频繁根据 key 查找应该用什么？

题目思想：如果需要频繁根据学号查找学生信息，是用 list 逐个查，还是用 dict 更合适？

前面已经学过：list 更偏向按位置 index 访问；list 查某个值通常是 O(n)；dict 根据 key 查找、赋值和 key in d 平均是 O(1)。

参考答案：用 dict 更合适。可以把学号作为 key、学生信息作为 value。这样查找时按 key 定位，平均复杂度是 O(1)；如果用 list 逐个找学号，通常需要线性扫描，平均复杂度是 O(n)。
